# 项目核心目录结构

本项目遵循 TRAE_VIBE_RULES.md 中定义的代码分层规范，核心目录结构如下：

```
lib/
├── api/
├── models/
├── pages/
├── provider/
└── utils/
```

## 各目录职责说明

### `api/` - 网络请求层

- **核心职责**: 统一管理应用的所有网络请求。
- **内容**:
    - **Dio 封装**: 包含一个单例的 Dio 实例，配置基础 URL、超时时间、拦截器（用于日志、Token 处理、错误处理等）。
    - **接口分类**: 根据后端模块（如 `user_api.dart`, `inquiry_api.dart`）将 API 请求分门别类，方便管理和调用。
- **原则**: 所有与后端 FastAPI 的交互都必须通过此目录中的方法进行。

### `models/` - 数据模型层

- **核心职责**: 定义与后端接口对应的 Dart 数据模型。
- **内容**:
    - **Null-Safe**: 所有模型必须严格遵循 null-safe 规范。
    - **序列化/反序列化**: 包含 `fromJson` 和 `toJson` 方法，用于 Dart 对象与 JSON 字符串之间的转换。
- **原则**: 确保前端模型与后端 FastAPI 的数据结构完全对齐，是数据流通的基石。

### `pages/` - 页面/UI 层

- **核心职责**: 存放应用的所有页面（UI 视图）。
- **内容**:
    - **页面组件**: 每个页面是一个独立的 Widget，负责UI的展示和用户交互。
    - **逻辑分离**: 页面本身应保持简洁，复杂的业务逻辑和状态管理应交由 `provider` 处理。
- **原则**: 聚焦于“看”和“用”，不做或少做数据处理。

### `provider/` - 状态管理层
- **核心职责**: 管理全局或模块级的可变状态（如适老化模式切换）。
- **技术选型**: 严格使用 `package:provider/provider.dart` 结合 `ChangeNotifier`。
- **核心文件**: 
    - `app_provider.dart`: 管理主题、字体大小、适老化开关。
- **原则**: 逻辑层通过 `notifyListeners()` 触发 UI 刷新，实现数据与视图解耦。

### `api/` - 网络请求层
- **内容**: 包含 Dio 单例封装，拦截器处理 Token 和错误码。
- **原则**: 所有的后端数据交换必须经过此层。

### `utils/` - 工具类层

- **核心职责**: 存放通用的工具、常量或扩展函数。
- **内容**:
    - **通用工具**: 如日期格式化、路由管理、设备信息获取等。
    - **常量定义**: 如应用的颜色、字符串、API 路径常量等。
- **原则**: 高内聚、低耦合，提供可在应用全局复用的功能。
